# MusicPlayer v0.4.0 音频引擎 · 设计文档

- 创建日期：2026-05-01
- 版本：v0.4.0
- 文档状态：Draft v1.0
- 基于：v0.3.0 library scanner（`main` merge commit `f188c81`，tag `v0.3.0` 已并入）

---

## 1. 背景

v0.3.0 已完成本地音乐库扫描：用户可以添加音乐文件夹，Rust 后台扫描 `mp3 / m4a / flac / wav / aac`，通过 `lofty` 读取标签、提取封面、计算 hash，并将曲目写入 SQLite。资料库现在有真实 `TrackView` 数据，但 `src-tauri/src/player/*`、`commands/player.rs`、前端 `usePlayer` 与 `MiniPlayer` 仍是 stub。

v0.4.0 的任务是把“资料库里有歌”推进到“真正能播放、能切歌、能被系统控制”。用户已明确要求一次做满，因此本版本同时覆盖：

1. Rust 侧 `symphonia + cpal` 真实播放
2. 播放状态机与队列
3. Gapless 双缓冲
4. macOS Now Playing / Control Center / 媒体键
5. 前端 `playerRepo + usePlayer + MiniPlayer` 真实连接

本设计遵守项目不可违反原则：**PCM 不过 IPC**。前端只发控制命令、接收状态事件；解码后的音频样本只在 Rust 进程内流向 `cpal` 输出设备。

## 2. 目标 / 非目标

### 2.1 目标

1. 用户从任意资料库列表双击/点击播放一首已扫描曲目，Rust 取 `tracks.file_path` 并输出到系统默认音频设备。
2. 支持播放 / 暂停 / 恢复 / 停止 / seek / 上一首 / 下一首。
3. 支持设置播放队列：当前曲目、即将播放、历史；保留随机和循环模式的状态结构。
4. 每 100ms 向前端发 `playback_progress`，曲目切换时发 `track_changed`，状态变化时发 `playback_state`，错误时发 `playback_error`。
5. 完播阈值沿用全局设计：播放达到曲目时长 95% 记为 completed，写 `play_history` 并累加 `tracks.play_count`。
6. Gapless：当前曲目播放到尾部前预解码下一首，下一首 PCM buffer 已就绪时做到无静音间隔切换。
7. macOS Now Playing：Control Center 显示标题、艺人、专辑、时长、进度、播放状态；媒体键能触发 play/pause/next/previous/seek。
8. 前端 `MiniPlayer` 展示当前曲目、播放状态、进度和基础控制；`usePlayer` 订阅 Rust 事件并同步 Zustand。
9. 所有核心状态和队列逻辑有 Rust 单元测试；前端 hook/store 有 Vitest 覆盖。
10. `pnpm tauri build --debug --bundles app` 生成可运行 `.app`，手测能播放真实扫描曲目。

### 2.2 非目标

- 不做均衡器、音频可视化、歌词、桌面歌词、外部音频设备选择 UI。
- 不做跨平台 Web 音频实现；v0.4.0 以 macOS Tauri 原生版验收。
- 不做完整播放列表编辑 UI；v0.4.0 只消费已有曲目/播放列表查询结果。
- 不做复杂 replay gain、响度标准化、交叉淡入淡出。
- 不承诺所有罕见 FLAC / ALAC 变体都可播放；不支持时返回 `AppError::Playback` 并继续队列。

## 3. 用户旅程

### 3.1 播放资料库中的一首歌

```
用户在 LibraryPage / Search / AlbumDetail 选择曲目
  │
  ▼
前端调用 playerRepo.play({ trackId, queueTrackIds, queueIndex })
  │
  ▼
Rust commands::player::play
  - 从 DB 读取 TrackView
  - 校验 missing_at / file_path 可访问
  - PlayerManager 发送 EngineCommand::LoadQueueAndPlay
  │
  ▼
AudioEngine 线程
  - symphonia probe + decoder
  - cpal default_output_device + output stream
  - ring buffer 向 audio callback 供样本
  │
  ▼
事件回前端
  - playback_state: Loading -> Playing
  - track_changed: 当前曲目元数据
  - playback_progress: position/duration/buffered
```

### 3.2 暂停、恢复、seek

- `pause()`：音频 callback 继续输出静音或暂停 stream，状态变为 `Paused`，进度冻结，Now Playing rate=0。
- `resume()`：从当前位置继续输出，状态变为 `Playing`，Now Playing rate=1。
- `seek(positionMs)`：发命令给 engine，decoder 使用 `FormatReader::seek` 跳转；seek 成功后清空当前 PCM buffer，重新填充；seek 期间状态为 `Buffering`。

seek 边界：

- `positionMs < 0`：clamp 到 0。
- `positionMs > durationMs`：clamp 到 `durationMs`。
- seek 成功后立即更新 `positionMs` 到目标值。
- seek 前是 `Paused`：seek 完成后仍回到 `Paused`，不自动恢复播放。
- seek 前是 `Playing`：seek 完成后继续 `Playing`。
- seek 清空当前 PCM ring buffer，并丢弃任何正在进行的 gapless 预解码结果。

### 3.3 自动下一首与 Gapless

- 队列中存在下一首时，当前曲目剩余约 1 秒或 PCM buffer 降到阈值时启动预解码。
- 预解码只读取下一首的开头 PCM，不接管队列状态。
- 当前曲目结束时：
  - 预解码 ready 且 sample rate / channel layout 与输出流兼容：立即切换 buffer，发 `track_changed`。
  - 预解码失败或格式不兼容：走普通 next 加载路径，并发 `playback_error` 的 warning 级事件，不终止整个播放器。

### 3.4 macOS 系统控制

- Rust `system::now_playing::NowPlaying` 只消费 `PlayerSnapshot`，不直接读数据库。
- `MPNowPlayingInfoCenter` 更新：
  - `MPMediaItemPropertyTitle`
  - `MPMediaItemPropertyArtist`
  - `MPMediaItemPropertyAlbumTitle`
  - `MPMediaItemPropertyPlaybackDuration`
  - `MPNowPlayingInfoPropertyElapsedPlaybackTime`
  - `MPNowPlayingInfoPropertyPlaybackRate`
- `MPRemoteCommandCenter` 注册 play / pause / toggle / next / previous / changePlaybackPosition 回调；回调只向 `PlayerManager` 发送命令。

## 4. 架构

### 4.1 模块拓扑

```
React
├── repositories/playerRepo.ts
├── hooks/usePlayer.ts
├── stores/playerStore.ts
└── components/player/MiniPlayer.tsx

             Tauri commands/events
                    │
                    ▼

Rust
├── commands/player.rs
│   └── thin command wrapper; no decode logic
├── player/
│   ├── manager.rs      # Tauri State; owns command channel + event bridge
│   ├── engine.rs       # audio thread; symphonia decode + cpal output
│   ├── decoder.rs      # file -> decoded PCM chunks; seek support
│   ├── queue.rs        # current/upcoming/history/repeat/shuffle
│   ├── state.rs        # state machine + serializable snapshots
│   └── gapless.rs      # next-track predecode buffer
├── system/now_playing.rs
└── db::{tracks, play_history}
```

### 4.2 线程模型

1. Tauri command 线程不解码、不阻塞播放。
2. `PlayerManager` 是 `tauri::State<PlayerManager>`，内部持有：
   - `command_tx: crossbeam_channel::Sender<PlayerCommand>`
   - `snapshot: Arc<parking_lot::Mutex<PlayerSnapshot>>`
   - `db: Database` clone，用于取曲目和写播放历史
   - `app: AppHandle`，只在 Manager / 普通 worker 线程 emit 事件
3. `AudioEngine` 独占一个 Rust worker thread：
   - 接收 `PlayerCommand`
   - 管理播放生命周期、decoder、PCM buffer、output stream、gapless buffer
   - 通过 `PlayerEvent` 向 `PlayerManager` 回传状态变化
   - 不直接持有 `AppHandle`，不直接 emit 前端事件
4. `cpal` audio callback 只做实时安全工作：
   - 从 PCM ring buffer 拿样本
   - 应用 volume
   - 应用 mute
   - buffer 不足时输出 0.0
   - 不做 DB、IPC、文件 IO、锁等待长任务

`AppHandle::emit` 不得出现在 cpal callback，也不得出现在任何实时音频路径里。所有 `playback_*` 事件由 `PlayerManager` 在普通线程根据 `PlayerEvent` 和 `PlayerSnapshot` 统一发出。

### 4.3 为什么不把播放写成 async command

播放不是一次性 RPC，而是长期运行的状态机。`async fn play()` 只负责发控制消息并快速返回；真正播放生命周期由 `AudioEngine` 线程管理。这样能避免 IPC command 被音频 IO、文件读取或 decoder seek 阻塞。

### 5.5 Tauri commands

`src-tauri/src/commands/player.rs`：

| Command | 参数 | 行为 |
|---|---|---|
| `player_play` | `{ trackId, queueTrackIds?, queueIndex? }` | 若传 queue 则按 queue 播放；否则单曲队列 |
| `player_pause` | none | 暂停 |
| `player_resume` | none | 恢复 |
| `player_toggle` | none | 播放/暂停切换 |
| `player_stop` | none | 停止并清 Now Playing |
| `player_seek` | `{ positionMs }` | 跳转 |
| `player_next` | none | 下一首 |
| `player_previous` | none | 上一首；若当前播放超过 3 秒，先回到 0 |
| `player_set_volume` | `{ volume }` | 0.0 到 1.0 |
| `player_set_muted` | `{ muted }` | 设置静音 |
| `player_toggle_mute` | none | 切换静音 |
| `player_get_state` | none | 返回 `PlayerSnapshot` |

旧的 `play/pause/resume/seek/next/prev` command 名称可以保留为兼容 wrapper，但前端 v0.4.0 应使用 `player_*` 命名，避免与未来库命令混淆。

`player_play` 参数校验：

1. `queueTrackIds` 未传或为空：构造单曲队列 `[trackId]`，`queueIndex = 0`。
2. `queueTrackIds` 已传且 `queueIndex` 合法：要求 `queueTrackIds[queueIndex] == trackId`。
3. 若 `queueTrackIds[queueIndex] != trackId`：返回 `AppError::InvalidInput("queueIndex does not point at trackId")`。
4. 若 `queueIndex` 越界：返回 `AppError::InvalidInput("queueIndex out of range")`。
5. 队列中的 missing track：自动跳过并发 recoverable `playback_error`。
6. 队列全部不可播放：状态进入 `Error`，保留队列，返回 `AppError::Playback`。

`player_set_volume` 对 `volume` 使用严格校验：`0.0..=1.0` 内接受；`NaN`、负数或大于 1.0 返回 `AppError::InvalidInput`，不隐式修正，避免前端状态和后端状态分叉。

### 5.6 Tauri events

| Event | Payload | 频率 |
|---|---|---|
| `playback_state` | `PlayerSnapshot` | 状态变化时 |
| `playback_progress` | `{ trackId, positionMs, durationMs }` | Playing 时 100ms 节流 |
| `track_changed` | `PlayerSnapshot` | 当前曲目变化时 |
| `playback_error` | `{ trackId?, message, recoverable }` | 错误时 |

`playback_error` 的 payload 使用可判别 code：

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
#[serde(rename_all = "camelCase")]
pub enum PlaybackErrorCode {
    FileNotFound,
    PermissionDenied,
    DecodeFailed,
    OutputUnavailable,
    StreamError,
    InvalidInput,
    Unknown,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
#[serde(rename_all = "camelCase")]

    pub track_id: Option<i64>,
    pub code: PlaybackErrorCode,
```

前端据此区分 UI：

- `decodeFailed`：toast + 自动下一首
- `outputUnavailable`：阻塞播放控件
- `invalidInput`：开发期错误，`console.error`
- `fileNotFound`：提示重新扫描资料库

## 6. 音频解码与输出

### 6.1 Decoder

`player/decoder.rs` 负责：

1. 打开本地文件并交给 `symphonia::default::get_probe()`。
2. 选择默认音轨或第一个可解码音轨。
3. 用 `symphonia::default::get_codecs().make()` 创建 decoder。
4. 每个 packet 解码为 `AudioBufferRef`。
5. 用 `SampleBuffer<f32>::copy_interleaved_ref()` 转成 interleaved `Vec<f32>`。
6. 输出 `DecodedChunk { samples, sample_rate, channels, frames, position_ms }`。

v0.4.0 统一内部 PCM 格式为 interleaved `f32`。如输出设备不是 f32，由 cpal callback 按 sample format 转换。

### 6.2 Output Stream

`player/engine.rs` 使用 cpal：

1. `cpal::default_host()`
2. `host.default_output_device()`
3. `device.default_output_config()`
4. 按 `SampleFormat::{F32, I16, U16}` 创建 output stream

默认使用设备配置，不在 v0.4.0 做设备选择 UI。若曲目采样率与设备采样率不同，v0.4.0 使用简单线性重采样 helper；后续可替换成更高质量 resampler。

默认输出设备异常恢复策略：

1. `default_output_device()` 不存在：`player_play` 返回 `AppError::Playback("no default output device")`，状态保持 `Idle` 或当前可恢复状态。
2. 播放中 cpal stream error：
   - emit `playback_error { code: "streamError", recoverable: true }`
   - 尝试重新打开 `default_output_device()` 一次
   - 成功：恢复播放，保留 queue 和 position
   - 失败：状态进入 `Error`，保留 queue 和 position，不写 completed
3. 用户恢复设备后点击 resume：重新初始化 output stream，并从当前 position 继续。

重采样和声道支持边界：

- 支持 `mono -> stereo`：复制单声道到左右声道。
- 支持 `stereo -> stereo`。
- 支持常见 `44100Hz -> 48000Hz` 和 `48000Hz -> 44100Hz` 线性重采样。
- 不支持 5.1 / 7.1 多声道 downmix；遇到非常规 channel layout 返回 recoverable decode/output error 并尝试下一首。
- 输出设备 sample format 支持 `F32 / I16 / U16`；其他格式返回 `AppError::Playback`。

### 6.3 Buffering

采用两层 buffer：

1. `DecodedChunk` 队列：decoder 线程写入，engine 消费。
2. `PcmRingBuffer`：cpal callback 读取固定数量样本。

水位策略：

- 高水位：约 500ms PCM，达到后 decoder 暂停填充。
- 低水位：约 150ms PCM，低于后 decoder 继续填充。
- underrun：输出静音，状态可短暂变为 `Buffering`，但不写播放历史。

## 7. Queue / Repeat / Shuffle

### 7.1 PlayQueue

`player/queue.rs`：

```rust
pub struct PlayQueue {
    items: Vec<i64>,
    index: usize,
    history: Vec<i64>,
    repeat_mode: RepeatMode,
    shuffle: bool,
    shuffled_order: Vec<usize>,
}
```

### 7.2 RepeatMode

```rust
#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq)]
#[serde(rename_all = "camelCase")]
pub enum RepeatMode {
    Off,
    All,
    One,
}
```

### 7.3 行为

- `next()`：
  - RepeatOne：返回当前 index
  - 有下一首：index + 1
  - 到队尾且 RepeatAll：回到 0
  - 到队尾且 RepeatOff：返回 None，状态 `Ended`
- `previous()`：
  - `PlayQueue` 本身不接收 position，也不知道播放进度
  - `PlayerManager` / `AudioEngine` 根据 snapshot 判断：当前 position > 3000ms 时执行 seek(0)，queue index 不变
  - 当前 position <= 3000ms 时才调用 `queue.previous()`
- `shuffle`：
  - v0.4.0 生成 `shuffled_order`，当前曲目保持不变，后续顺序打乱
  - 关闭 shuffle 时恢复原 queue index

`PlayQueue` 只负责 index、history、repeat、shuffle，不读数据库、不判断文件是否 missing、不处理播放位置。missing track 跳过逻辑属于 `PlayerManager`，因为它需要 DB 查询和错误事件。

## 7.4 PlaybackSession / 播放历史

播放历史不能只靠 position 判断，否则用户 seek 到末尾会刷播放次数。每次开始播放一个 track 时创建独立 session：

```rust
pub struct PlaybackSession {
    pub id: String,
    pub track_id: i64,
    pub started_at_ms: i64,
    pub max_position_ms: i64,
    pub real_played_ms: i64,
    pub completed_written: bool,
}
```

写入规则：

1. 同一个 `play_session_id` 只允许写一次 completed。
2. seek 到 95% 不立刻算 completed。
3. completed 同时满足：
   - `max_position_ms >= duration_ms * 0.95`
   - `real_played_ms >= min(30_000ms, duration_ms * 0.5)`
4. 用户手动 next/stop 时，如果当前 session 满足 completed 条件，先 flush 当前 session。
5. decode error / file missing / permission error / output device error 不写 completed。
6. 自动切歌时先 flush 当前 session，再创建下一首 session。

## 8. Gapless 设计

### 8.1 边界

Gapless 只处理“下一首开头 PCM 已准备好并能立即切换”的问题，不管理播放队列、不写历史、不发 UI 事件。它消费 `QueuePreview { current_track_id, next_track_id }`，产出 `PredecodedTrack`。

### 8.2 PredecodedTrack

```rust
pub struct PredecodedTrack {
    pub track_id: i64,
    pub sample_rate: u32,
    pub channels: u16,
    pub start_samples: Vec<f32>,
    pub decoder: Decoder,
}
```

### 8.3 触发条件

- 当前曲目剩余时长 <= 1000ms，且 queue.next() 存在。
- 当前曲目剩余 PCM frames <= 2 * device buffer size。
- 当前没有正在预解码的同一首 next track。

### 8.4 成功标准

自动切歌时：

1. 不停止 cpal stream。
2. callback 从当前 ring buffer 读完后继续读 next `start_samples`。
3. engine 切换 current decoder 到预解码 decoder。
4. 同步发 `track_changed` 与 Now Playing 更新。

### 8.5 降级

- next track decode 失败：记录 recoverable `playback_error`，普通加载下一首。
- sample rate / channels 不兼容且 v0.4.0 resampler 无法处理：普通加载下一首。
- 用户在预解码期间 seek/next/previous：丢弃预解码结果。

## 9. macOS Now Playing

### 9.1 模块职责

`src-tauri/src/system/now_playing.rs` 提供：

```rust
pub struct NowPlayingController;

impl NowPlayingController {
    pub fn new(command_tx: Sender<PlayerCommand>) -> AppResult<Self>;
    pub fn update(&self, snapshot: &PlayerSnapshot) -> AppResult<()>;
    pub fn clear(&self);
}
```

### 9.2 安全边界

`objc2-media-player` 调用集中在 `system/now_playing.rs`。其他模块不得直接依赖 Objective-C 类型，避免 unsafe 扩散。

### 9.3 命令回调

Remote command 回调只做一件事：把系统命令转换成 `PlayerCommand`。不在回调里读 DB、不 decode、不 emit 前端事件。

### 9.4 平台条件

macOS 专属代码使用 `#[cfg(target_os = "macos")]`。非 macOS 使用 no-op `NowPlayingController`，保持 `cargo test` 可在未来 CI 矩阵中运行。

```ts
  status: PlaybackStatus;
  current: NowPlayingTrack | null;
  positionMs: number;
  durationMs: number;
  volume: number;
  muted: boolean;
  queueIndex: number | null;
  queueLen: number;
  repeatMode: RepeatMode;
  shuffle: boolean;
}

export interface PlayParams {
  trackId: number;
  queueTrackIds?: number[];
  queueIndex?: number;
}

export const playerRepo = {
  play: (params: PlayParams) => invoke<void>("player_play", params),
  pause: () => invoke<void>("player_pause"),
  resume: () => invoke<void>("player_resume"),
  toggle: () => invoke<void>("player_toggle"),
  stop: () => invoke<void>("player_stop"),
  seek: (positionMs: number) => invoke<void>("player_seek", { positionMs }),
  next: () => invoke<void>("player_next"),
  previous: () => invoke<void>("player_previous"),
  setVolume: (volume: number) => invoke<void>("player_set_volume", { volume }),
  setMuted: (muted: boolean) => invoke<void>("player_set_muted", { muted }),
  toggleMute: () => invoke<void>("player_toggle_mute"),
  getState: () => invoke<PlayerSnapshot>("player_get_state"),
};
```

`repositories/` 仍是唯一允许调用 Tauri command 的前端目录。

### 10.2 playerStore

Zustand store 存：

- `status`
- `current`
- `positionMs`
- `durationMs`
- `volume`
- `muted`
- `queueIndex`
- `queueLen`
- `repeatMode`
- `shuffle`
- `lastError`

### 10.3 usePlayer

`usePlayer()`：

1. mount 时调用 `player_get_state` 初始化。
2. 订阅 `playback_state`、`playback_progress`、`track_changed`、`playback_error`。
3. 暴露 command wrapper。
4. unmount 时清理 Tauri listener。

前端更新约束：

- `playback_progress` 只更新 `positionMs / durationMs`。
- `track_changed` 和 `playback_state` 才更新 `current / status / queueIndex / queueLen / repeatMode / shuffle` 等结构性状态。
- `MiniPlayer` 使用 Zustand selector 只订阅需要的 slice，避免 100ms progress 事件触发全应用重渲染。
- 用户拖动 seek bar 时使用本地 optimistic position；拖动中不被 100ms progress 覆盖，commit 时调用 `player_seek`。

### 10.4 MiniPlayer

v0.4.0 MiniPlayer 必须包含：

- 当前标题 / 艺人
- 播放 / 暂停按钮
- 上一首 / 下一首按钮
- 进度条 + 时间
- 音量 slider
- 错误状态短提示

视觉保持现有简洁结构，不在 v0.4.0 做 Now Playing 大屏或复杂动画。

## 11. 错误处理

| 场景 | Rust error | 前端表现 | 队列行为 |
|---|---|---|---|
| track id 不存在 | `AppError::NotFound` | toast / inline error | 不改变当前播放 |
| 文件缺失 | `AppError::FileNotFound` | 标记错误 | 自动尝试下一首 |
| 无权限 | `AppError::PermissionDenied` | 提示重新添加文件夹 | 自动尝试下一首 |
| 解码失败 | `AppError::Playback` | 显示格式不支持 | 自动尝试下一首 |
| 输出设备不可用 | `AppError::Playback` | 阻塞播放控件 | 保持队列 |
| cpal stream error | `playback_error` | 提示并停止 | 状态 Error |
| Now Playing 更新失败 | log + recoverable error | 不阻塞播放 | 不影响队列 |

## 11.1 App 退出生命周期

Tauri 退出 / window close 时：

1. `PlayerManager` 发送 `PlayerCommand::Shutdown`。
2. `AudioEngine` 停止 decoder，并停止预解码任务。
3. drop cpal stream。
4. `NowPlayingController::clear()` 清空系统媒体栏信息。
5. flush 当前 `PlaybackSession`；只有满足 completed 条件才写 completed。
6. 注销 remote command callbacks，释放线程资源。

Shutdown 必须幂等：重复调用不 panic，不重复写播放历史。

## 12. 测试策略

### 12.1 Rust 单元测试

- `queue.rs`
  - next/previous/repeat-one/repeat-all/shuffle
  - previous 在 position > 3s 时返回 seek-to-zero 决策
  - `queueTrackIds + queueIndex` mismatch 返回 `InvalidInput`
  - repeatAll 到队尾回到 0
  - shuffle 开启后 current 不变
  - shuffle 关闭后恢复原 index
- `state.rs`
  - 合法状态转换
  - snapshot serde camelCase
- `session.rs` 或 `manager.rs`
  - seek 到 95% 不写 completed
  - 同一 `play_session_id` 不重复写 play_count
  - next 时满足 completed 条件会 flush
  - decode error 不写 completed
- `decoder.rs`
  - 使用 v0.3.0 fixtures 解码 mp3/flac/m4a/wav/no-tag
  - 输出 sample_rate/channels/duration 非零
  - seek 到 500ms 后继续产出 PCM
- `gapless.rs`
  - 预解码成功
  - 用户 skip 后丢弃过期预解码
  - decode error 降级为普通 next
- `manager.rs`
  - command channel 更新 snapshot
  - 完播 95% 写 play_history
  - Paused 状态 seek 后仍 Paused
  - Playing 状态 seek 后继续 Playing
  - buffer underrun 输出静音但不 panic
  - output stream error 重试一次并保留 queue/position

### 12.2 前端测试

- `playerRepo` command name + params
- `playerStore` reducer/state update
- `usePlayer` event subscription
- `MiniPlayer` 根据 Playing/Paused/Error 显示正确 controls
- `playback_progress` 只更新 position

## 13. 风险与缓解

| 风险 | 描述 | 缓解 |
|---|---|---|
| cpal callback 中误做阻塞工作 | 造成卡顿或音频断裂 | callback 只读 ring buffer；IO/DB/IPC 全在 engine 线程 |
| sample rate 不匹配 | 不同文件和设备配置可能不一致 | v0.4.0 加简单线性重采样；失败时普通切歌 |
| Gapless 复杂度 | 双缓冲和 seek/skip 交互容易出错 | Gapless 模块只持有 next predecode；任何用户控制命令都可丢弃它 |
| Now Playing unsafe 扩散 | Objective-C bridge 易污染架构 | unsafe 只在 `system/now_playing.rs`，对外暴露安全 Rust API |
| 媒体键回调线程不确定 | 可能从非 engine 线程进入 | 回调只 send command，不直接改状态 |
| 播放历史重复写 | pause/seek/next 都可能触发边界 | 每个 track session 分配 `play_session_id`，只在 session end 写一次 |
| 前端进度更新抖动 | 100ms progress 事件触发过多组件渲染 | progress 只更新 position slice；拖动 seek 时本地 optimistic |
| 队列参数不一致 | `trackId` 和 `queueIndex` 指向不同曲目 | command 层返回 `InvalidInput`，不隐式纠正 |
| 退出时资源悬挂 | cpal stream 或 remote command callback 未释放 | Tauri 退出时发送幂等 Shutdown |
| 主工作树 Cargo target 污染 | v0.3.0 merge 验证发现 `muda` rlib 缓存问题 | v0.4.0 验证使用 `/tmp/musicplayer-v040-target` 或清理 target 后跑 |
