# MusicPlayer v0.8.0 质量加固设计文档

> 版本：v0.8.0 v2（Review 后修订）
> 日期：2026-05-02
> 状态：待 Review

## 1. 背景与目标

v0.7.0 已交付 v1.0 全部功能（扫描、播放、搜索、收藏、播放列表、Now Playing、快捷键、i18n、主题）。v0.8.0 的目标是从"功能完整"提升到"可发布质量"。

**成功标准**（可测量、可验收）：

1. **测试覆盖**：前端测试数 ≥ 80，且关键模块（TrackTable、SearchPage、FavoritesPage、RecentPlaysPage、NowPlayingOverlay、useToggleFavoriteMutation）的有效交互断言覆盖率 ≥ 80%。禁止无断言或仅测 className 的弱测试。
2. **E2E 链路**：Playwright 覆盖 5 条核心用户链路，每条独立初始化测试数据，无级联依赖。
3. **首屏加载**：production Tauri debug build（`pnpm tauri build --debug`）下，主窗口可交互时间 P75 < 2s。本地机器记录 5 次取中位数。
4. **大列表滚动**：1000 曲目 SongsPage 启用虚拟滚动后，DOM 中实际 track row 数量 ≤ 可视区 + overscan（≤ 80），无明显输入阻塞。
5. **可访问性**：Lighthouse Accessibility ≥ 90；键盘可达、焦点不逃逸、核心控件有语义标签。

**不做**：
- 100% 测试覆盖率
- 性能基准测试自动化
- 完整屏幕阅读器体验（推迟到 v1.1）
- 高对比度主题（推迟到 v1.1）
- WCAG AAA 级修复（推迟到 v1.1）
- React Query 缓存命中率量化（除非专门埋点，否则删除此指标）

## 2. 执行顺序

**非严格串行**，按以下 5 个 Phase 推进，每个 Phase 独立验收：

| Phase | 内容 | 依赖 |
|---|---|---|
| 0 | 质量基线记录 | 无 |
| 1 | 单元/组件测试补齐 | Phase 0 |
| 2 | E2E Spike + 基础设施 | Phase 0 |
| 3 | 性能与可访问性修复 | Phase 1（测试保护） |
| 4 | 回归门禁 + E2E 完整链路 | Phase 1-3 |

Phase 1 和 Phase 2 可并行启动（测试补齐不依赖 E2E），但 Phase 3 必须在 Phase 1 之后（性能/可访问性改动需要测试保护）。

### 4.3 测试策略

- 使用 `vi.mock` 隔离 Tauri IPC（已有模式）
- 优先使用 `userEvent.setup()` 模拟真实交互
- 异步 UI 使用 `findBy*` / `waitFor` 等待元素出现/消失
- 只有在直接操作 store、timer 或外部订阅时才手动使用 `act`
- 不写快照测试
- 禁止无断言测试和只测 className 的弱测试
- 测试文件与源文件同目录

### 4.4 验收标准

- `npx vitest run` 全部通过，0 失败、0 unhandled rejection
- 测试数 ≥ 80
- `pnpm build` 通过
- `pnpm lint` 通过

## 5. Phase 2：E2E Spike + 基础设施

### 5.1 技术方案

Playwright + Tauri WebDriver（`tauri-driver`）。

### 5.2 Spike 阶段（必须先通过）

在写任何 E2E 链路前，先验证基础设施：

1. 安装 `tauri-driver`，验证可启动 debug app
2. 写一条最小测试：启动 App → 定位主窗口 → 断言 AppShell 可见
3. 在本机跑通后再接 CI
4. Spike 通过后再扩展 5 条链路

如果 Spike 失败（WebDriver 无法稳定启动），则 E2E 降级为 Vitest 集成测试（不启动真实 Tauri 窗口）。

### 5.3 测试数据与隔离策略

- 每条 E2E 使用独立 `app_data_dir`，避免数据库互相污染
- 每条 E2E 启动前清理 SQLite、封面缓存、设置文件
- fixture 音频目录只读，不在测试中修改
- 播放列表、收藏、最近播放通过 seed 生成，每条测试独立
- CI 中缓存 ffmpeg 生成产物，不缓存数据库

**E2E 测试模式**：
- 启动 App 时设置 `MUSICPLAYER_E2E=1` 环境变量
- 后端允许使用固定 fixture 路径绕过系统原生目录选择器
- 或提供仅测试环境可用的 seed command：`e2e_seed_library(fixturePath)`
- 避免依赖系统原生 dialog（Playwright 无法稳定操作）

### 5.4 五条核心链路

#### 链路 1：扫描 → 浏览
1. 通过 seed command 注入测试 fixture 路径（绕过系统 dialog）
2. 触发扫描
3. 等待扫描完成事件
4. 导航到"歌曲"页面
5. 验证曲目列表非空

#### 链路 2：播放控制
1. 导航到"歌曲"页面
2. 点击第一首曲目的播放按钮
3. 验证 Mini Player 显示曲目信息
4. 点击暂停按钮
5. 点击下一首按钮
6. 验证曲目信息变更

#### 链路 3：搜索 → 收藏
1. 按 Cmd+F 导航到搜索页
2. 输入搜索词
3. 等待搜索结果
4. 点击第一首曲目的收藏按钮
5. 导航到"收藏"页面
6. 验证曲目出现在收藏列表

#### 链路 4：播放列表 CRUD
1. 导航到"播放列表"页面
2. 点击"创建"按钮
3. 输入播放列表名称
4. 确认创建
5. 添加曲目到播放列表
6. 验证曲目出现在播放列表

#### 链路 5：Now Playing
1. 播放一首曲目
2. 点击 Mini Player 的曲目区域
3. 验证 Now Playing 大屏打开
4. 验证曲目信息、进度条、控制按钮可见
5. 按 Esc 关闭
6. 验证 Now Playing 关闭

### 5.5 约束

- E2E 只测核心链路，不追求全面覆盖
- 单元测试已覆盖的逻辑不重复测
- 每条链路独立，不依赖其他链路的执行顺序

### 5.6 验收标准

- Spike 通过：最小测试在本机稳定运行
- `pnpm test:e2e` 全部通过
- 每条链路执行时间 < 30s
- 测试可在 CI 中稳定运行（无 flaky）

## 6. Phase 3：性能与可访问性修复

### 6.1 性能优化

| 区域 | 问题 | 方案 | 验收标准 |
|---|---|---|---|
| TrackTable 大列表 | 1000+ 行 DOM 节点多 | 虚拟滚动（`@tanstack/react-virtual`） | DOM row 数 ≤ 80 |
| React Query 缓存 | 重复查询浪费 | 检查 `staleTime`，合理设置缓存过期 | 页面切换不重复触发相同请求 |
| 首屏 bundle | JS 包体积 | 检查 tree-shaking，动态导入大依赖 | bundle 体积不显著增加 |
| Rust 扫描 | 大库扫描慢 | 检查批量插入事务、FTS5 索引更新策略 | 1000 曲目扫描 < 10s |

### 6.2 可访问性修复

**v0.8 范围（AA 阻塞项）**：
- 键盘可达：所有交互元素可 Tab 到达
- 焦点顺序：从左到右、从上到下
- Now Playing 焦点陷阱：打开时焦点不逃逸
- 表单/按钮 `aria-label`：核心控件有语义标签
- 文本对比度 AA：正文 ≥ 4.5:1，大文本 ≥ 3:1
- UI 控件非纯颜色表达：收藏用心形图标+颜色，缺失用文字+颜色

**不承诺**：
- 完整屏幕阅读器体验（推迟到 v1.1）
- 高对比度主题（推迟到 v1.1）
- WCAG AAA 级修复（推迟到 v1.1）
- `aria-live` 播放状态变更（推迟到 v1.1，属于屏幕阅读器深度适配）

### 6.3 工具

- Chrome DevTools Lighthouse 可访问性审计
- axe DevTools 浏览器扩展
- 键盘手动测试（Tab/Enter/Esc/Space/方向键）
- Chrome DevTools Performance 面板记录滚动帧率
- `npx vite-bundle-visualizer` 检查 bundle 组成

### 6.4 验收标准

- 1000 曲目列表：DOM row ≤ 80，无明显输入阻塞
- Lighthouse Accessibility ≥ 90
- 所有交互元素可通过键盘到达
- Now Playing 打开时焦点不逃逸
- 首屏加载时间不退化
- `pnpm build` 产物体积不显著增加

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Playwright + Tauri WebDriver 配置复杂 | E2E 阶段阻塞 | Spike 先验证，失败则降级为 Vitest 集成测试 |
| 系统原生 dialog 无法自动化 | 链路 1 flaky | E2E 测试模式绕过系统 dialog |
| 虚拟滚动引入新 bug | 回归风险 | 充分测试后才提交，保留非虚拟模式降级 |
| 可访问性改动影响视觉 | UI 退化 | 改动后截图对比，确保视觉一致 |
| 1000 曲目测试数据生成慢 | 测试效率 | 用 ffmpeg 批量脚本，CI 中缓存 |
| 虚拟滚动改 TrackTable 导致测试返工 | 测试维护成本 | Phase 3 后补回测试，不在 Phase 1 写死 DOM 结构假设 |
