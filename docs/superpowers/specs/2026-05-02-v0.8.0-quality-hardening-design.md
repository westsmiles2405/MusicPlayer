# MusicPlayer v0.8.0 质量加固设计文档

> 版本：v0.8.0
> 日期：2026-05-02
> 状态：待 Review

## 1. 背景与目标

v0.7.0 已交付 v1.0 全部功能（扫描、播放、搜索、收藏、播放列表、Now Playing、快捷键、i18n、主题）。v0.8.0 的目标是从"功能完整"提升到"可发布质量"。

**成功标准**：
- 前端测试从 43 → 80+ 测试
- Playwright E2E 覆盖 5 条核心用户链路
- 首屏加载 < 2s，大库（1000+ 曲目）滚动流畅
- WCAG 2.1 AA 基本达标

**不做**：
- 100% 测试覆盖率
- 性能基准测试自动化
- 屏幕阅读器深度适配（推迟到 v1.1）
- 高对比度主题（推迟到 v1.1）

## 2. 执行顺序

严格顺序：测试覆盖率 → E2E → 性能 → 可访问性。每阶段独立验收后才进下一阶段。

## 4. 阶段二：E2E 测试

### 4.1 技术方案

Playwright + Tauri WebDriver（`tauri-driver`）。

**基础设施**：
- `tests/e2e/` 目录存放 Playwright 测试
- `playwright.config.ts` 配置 Tauri WebDriver
- `pnpm test:e2e` 命令
- CI 中先 `pnpm tauri build --debug` 再跑 E2E

### 4.2 五条核心链路

#### 链路 1：扫描 → 浏览
1. 打开设置页
2. 添加测试音乐文件夹
3. 触发扫描
4. 等待扫描完成事件
5. 导航到"歌曲"页面
6. 验证曲目列表非空

#### 链路 2：播放控制
1. 导航到"歌曲"页面
2. 点击第一首曲目的播放按钮
3. 验证 Mini Player 显示曲目信息
4. 点击暂停按钮
5. 点击下一首按钮
6. 验证曲目信息变更

#### 链路 3：搜索 → 收藏
1. 按 Cmd+F 导航到搜索页
2. 输入搜索词
3. 等待搜索结果
4. 点击第一首曲目的收藏按钮
5. 导航到"收藏"页面
6. 验证曲目出现在收藏列表

#### 链路 4：播放列表 CRUD
1. 导航到"播放列表"页面
2. 点击"创建"按钮
3. 输入播放列表名称
4. 确认创建
5. 添加曲目到播放列表
6. 验证曲目出现在播放列表

#### 链路 5：Now Playing
1. 播放一首曲目
2. 点击 Mini Player 的曲目区域
3. 验证 Now Playing 大屏打开
4. 验证曲目信息、进度条、控制按钮可见
5. 按 Esc 关闭
6. 验证 Now Playing 关闭

### 4.3 约束

- E2E 只测核心链路，不追求全面覆盖
- 单元测试已覆盖的逻辑不重复测
- 每条链路独立，不依赖其他链路的执行顺序
- 测试数据用 ffmpeg 生成的静音文件（已有 fixture）

### 4.4 验收标准

- `pnpm test:e2e` 全部通过
- 每条链路执行时间 < 30s
- 测试可在 CI 中稳定运行（无 flaky）

## 5. 阶段三：性能优化

### 5.1 目标指标

| 指标 | 目标 |
|---|---|
| 首屏加载（dev 模式） | < 2s |
| 1000+ 曲目列表滚动 | 60fps |
| React Query 缓存命中率 | > 90% |

### 5.2 优化方向

| 区域 | 问题 | 方案 |
|---|---|---|
| TrackTable 大列表 | 1000+ 行 DOM 节点多 | 虚拟滚动（`@tanstack/react-virtual`） |
| React Query 缓存 | 重复查询浪费 | 检查 `staleTime` 配置，合理设置缓存过期 |
| 首屏 bundle | JS 包体积 | 检查 tree-shaking，动态导入大依赖 |
| Rust 扫描 | 大库扫描慢 | 检查批量插入事务、FTS5 索引更新策略 |

### 5.3 验证方式

- Chrome DevTools Performance 面板记录滚动帧率
- `npx vite-bundle-visualizer` 检查 bundle 组成
- 手动测试 1000 曲目场景（用 ffmpeg 批量生成测试文件）
- 对比优化前后 Lighthouse Performance 分数

### 5.4 验收标准

- 1000 曲目列表滚动无卡顿
- 首屏加载时间不退化
- `pnpm build` 产物体积不显著增加

## 6. 阶段四：可访问性审计

### 6.1 审计范围

| 区域 | 审计点 | 修复方案 |
|---|---|---|
| 键盘导航 | 所有交互元素可 Tab 到达 | 检查 `tabIndex`、`focus()` 管理 |
| 焦点管理 | Now Playing 打开时焦点陷阱 | `focus-trap-react` 或手动实现 |
| 颜色对比度 | 文字/背景对比度 ≥ 4.5:1 | Chrome DevTools 检查，调整颜色 |
| ARIA 标签 | 所有按钮/控件有语义化 `aria-label` | 检查并补充缺失的 ARIA 属性 |
| 屏幕阅读器 | 核心操作可被朗读 | `aria-live` 区域用于播放状态变更 |
| 颜色盲友好 | 不仅靠颜色传达信息 | 收藏用心形图标+颜色，缺失用文字+颜色 |

### 6.2 工具

- Chrome DevTools Lighthouse 可访问性审计
- axe DevTools 浏览器扩展
- 键盘手动测试（Tab/Enter/Esc/Space/方向键）

### 6.3 约束

- 不做完整的屏幕阅读器适配（推迟到 v1.1）
- 不做高对比度主题（推迟到 v1.1）
- 只修复 AA 级阻塞问题，AAA 推迟

### 6.4 验收标准

- Lighthouse Accessibility 评分 ≥ 90
- 所有交互元素可通过键盘到达
- 焦点顺序符合逻辑（从左到右、从上到下）
- Now Playing 打开时焦点不逃逸

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Playwright + Tauri WebDriver 配置复杂 | E2E 阶段阻塞 | 先写 1 条链路验证可行，再扩展 |
| 虚拟滚动引入新 bug | 回归风险 | 充分测试后才提交，保留非虚拟模式降级 |
| 可访问性改动影响视觉 | UI 退化 | 改动后截图对比，确保视觉一致 |
| 1000 曲目测试数据生成慢 | 测试效率 | 用 ffmpeg 批量脚本，CI 中缓存 |
