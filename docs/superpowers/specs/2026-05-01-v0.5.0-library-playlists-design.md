# MusicPlayer v0.5.0 资料库浏览与播放列表 · 设计文档

- 创建日期：2026-05-01
- 版本：v0.5.0
- 文档状态：Draft v1.0
- 基于：v0.4.0 audio engine（`main` commit `0f9507f`，tag `v0.4.0` 已并入）

---

## 1. 背景

v0.3.0 已完成本地资料库扫描，v0.4.0 已完成 Rust 侧 `symphonia + cpal` 音频引擎、播放队列、MiniPlayer、Gapless 和 macOS Now Playing。当前主干的后端数据接口已经覆盖曲目、专辑、艺人、播放列表、搜索、播放历史和收藏；前端仍主要停留在一个统计型 `LibraryPage`，顶部只有“资料库 / 设置”两个链接。

v0.5.0 的目标是把 MusicPlayer 从“能扫描、能播放”推进到“能日常浏览和整理本地音乐”。本版本优先补齐 Apple Music 风格桌面资料库的基础信息架构：侧边栏、资料库分类、歌曲/专辑/艺人页面、专辑/艺人详情页、播放列表 CRUD、播放列表详情和排序，并确保所有列表都能使用 v0.4.0 的队列播放能力。

用户要求后续“都要”：资料库、播放列表、搜索、收藏/最近播放、Now Playing 大屏、毛玻璃视觉、快捷键、i18n、主题跟随系统都要进入产品路线。为了保持每个版本可验收，路线拆分为：

1. v0.5.0：资料库浏览 + 播放列表 MVP。
2. v0.6.0：全局搜索 + 收藏 + 最近播放。
3. v0.7.0：Now Playing 大屏 + Apple Music 风格视觉 polish + 快捷键/主题/i18n。

## 2. 目标 / 非目标

### 2.1 目标

1. 用真正的 `AppShell` 替换当前顶栏导航：左侧边栏、主内容区、底部 MiniPlayer。
2. 资料库侧边栏包含“最近添加、歌曲、专辑、艺人”和“播放列表”分组。
3. 支持资料库浏览页：
   - 最近添加：按添加时间倒序展示曲目。
   - 歌曲：表格展示全部曲目，支持按标题、艺人、专辑、添加时间排序。
   - 专辑：网格展示专辑封面、标题、艺人、曲目数。
   - 艺人：列表展示艺人，进入后可查看该艺人的曲目。
4. 支持详情页：
   - 专辑详情页展示封面、标题、艺人、年份、曲目列表。
   - 艺人详情页展示艺人名称、曲目数、专辑入口和曲目列表。
5. 支持播放列表 MVP：
   - 创建播放列表。
   - 重命名播放列表。
   - 删除播放列表。
   - 进入播放列表详情。
   - 从歌曲列表或详情页把曲目加入播放列表。
   - 从播放列表移除曲目。
   - 在播放列表详情中拖拽排序。
6. 补齐 v0.5.0 必需的 repository / Tauri command 契约，特别是详情查询和播放列表曲目 position。
7. 从任意曲目列表播放时，调用 `player_play(trackId, queueTrackIds, queueIndex)`，把当前可播放列表作为队列传给音频引擎。
8. 页面要覆盖空状态、加载状态、错误状态、missing 曲目状态和扫描完成后的 React Query 刷新。
9. 基础 Apple Music 风格视觉：毛玻璃侧边栏和 MiniPlayer 容器、封面缩略图、紧凑表格、清晰选中态、深浅色跟随系统。
10. 前端新增组件和页面需要 Vitest 覆盖关键交互；新增/变更 Rust 数据接口时必须补对应 Rust 测试。

### 2.2 非目标

- 不做 Now Playing 大屏；MiniPlayer 可保留点击入口占位，但不打开完整大屏。
- 不做完整全局搜索页面；搜索能力在 v0.6.0 实现。
- 不做收藏独立页和最近播放独立页；`setFavorite` 和 `historyRepo` 保持可用，但不作为主导航完成项。
- 不做歌词、动态封面背景、复杂转场动画、队列面板、均衡器。
- 不新增远程服务、账号、订阅、DRM、云同步。
- 不改 Rust 音频引擎架构，不让 PCM 通过 IPC。

## 3. 信息架构

### 3.1 路由

使用 `react-router` 的普通嵌套路由，`AppShell` 作为根布局。

```text
/
├── /library/recent        最近添加
├── /library/songs         歌曲
├── /library/albums        专辑
├── /library/albums/:id    专辑详情
├── /library/artists       艺人
├── /library/artists/:id   艺人详情
├── /playlists             播放列表总览
├── /playlists/:id         播放列表详情
└── /settings              设置
```

`/` 重定向到 `/library/recent`。未知路由进入 `/library/recent`，不新增独立 404 页面。

### 3.2 侧边栏

侧边栏分为三块：

1. 主导航：资料库分组。
2. 播放列表分组：展示现有播放列表，包含创建按钮。
3. 设置入口。

侧边栏不直接调用 Tauri。它只通过 React Query 调用 `playlistRepo.list()`，创建播放列表由弹窗组件回调 `playlistRepo.create()` 后刷新 query。

### 3.3 主内容区

主内容区保持页面级滚动，`MiniPlayer` 固定在底部。列表页不要让 body 滚动，避免 MiniPlayer 与内容重叠。

页面共同规则：

- 顶部标题区显示页面标题和主要操作。
- 加载时显示骨架或紧凑 loading。
- 错误时显示错误消息和重试按钮。
- 空状态必须给出下一步动作，例如“添加音乐文件夹”或“向播放列表添加歌曲”。

### 4.1 布局组件

重写现有 `AppShell`，并新增其余布局组件：

```text
src/components/layout/AppShell.tsx
src/components/layout/Sidebar.tsx
src/components/layout/PageHeader.tsx
src/components/layout/EmptyState.tsx
src/components/layout/LoadingState.tsx
src/components/layout/ErrorState.tsx
```

`AppShell` 负责总体结构：

- 左侧 `Sidebar`
- 中间 `<Outlet />`
- 底部 `MiniPlayer`
- 全局 `ScanProgressBar`

`Sidebar` 只管理导航选中态和播放列表入口，不承担页面数据加载之外的复杂业务。

### 4.2 资料库组件

新增：

```text
src/components/library/TrackTable.tsx
src/components/library/AlbumGrid.tsx
src/components/library/ArtistList.tsx
src/components/library/AddToPlaylistMenu.tsx
src/components/library/CoverArt.tsx
```

`TrackTable` 拆成容器和展示两层，避免把播放、播放列表、拖拽等业务继续堆进一个表格文件：

- `TrackTableView`：纯展示，接收 rows 和行事件回调，不调用 `usePlayer()`、不调用 repo。
- `TrackTable`：容器组件，接收 `tracks`，组装播放队列、加入播放列表菜单、移除、拖拽事件。

`TrackTable` 接收：

```ts
interface TrackTableProps {
  tracks: Track[] | PlaylistTrack[];
  queueContext: "recent" | "songs" | "album" | "artist" | "playlist";
  playlistId?: number;
  onAddToPlaylist?: (track: Track, playlistId: number) => Promise<void>;
  onRemoveFromPlaylist?: (track: PlaylistTrack) => Promise<void>;
  onReorderPlaylist?: (
    sourcePosition: number,
    destinationPosition: number,
  ) => Promise<void>;
}
```

`TrackTable` 内部可以调用 `usePlayer()`，因为它是用户交互容器；但它不得直接调用 `playlistRepo` 或任何 `invoke()`。添加到播放列表、移除、排序必须通过页面传入的回调完成。`TrackTableView` 不知道播放列表 repo、React Query 或 Tauri。

播放规则：

```ts
const queueTrackIds = tracks.map((track) => track.id);
const queueIndex = index;
await player.play(track.id, queueTrackIds, queueIndex);
```

如果 `track.missingAt !== null`，该行不可播放，不进入 `queueTrackIds`，播放按钮禁用。

`AlbumGrid` 和 `ArtistList` 只负责展示和导航，不负责播放。

### 4.3 播放列表组件

新增：

```text
src/components/playlists/PlaylistCreateDialog.tsx
src/components/playlists/PlaylistRenameDialog.tsx
src/components/playlists/PlaylistDeleteDialog.tsx
```

对话框使用原生 React 状态和现有样式，不引入新的 modal 依赖。删除播放列表必须二次确认，确认文案包含播放列表名称。

播放列表排序使用 HTML Drag and Drop 实现，避免 v0.5.0 引入额外拖拽依赖。拖拽只在播放列表详情页启用；其它资料库列表不支持拖拽排序。

拖拽规则：

- 空播放列表和单曲播放列表不显示拖拽 handle。
- 拖拽时显示目标行上方或下方的 insertion indicator。
- `Escape` 或 drop 到列表外取消拖拽，不调用 repo。
- 排序 mutation pending 时禁用新的拖拽，避免连续快速拖拽基于旧顺序计算。
- 排序失败后刷新服务端顺序，不保留 optimistic 假顺序。

## 6. 数据流与缓存

### 6.1 Repository API 对照表

实现计划第一步必须做 repository API audit。当前代码库已经有大部分 db 函数和 repo 方法，但 v0.5.0 仍有明确缺口。

| 前端方法 | Tauri command | 当前状态 | v0.5.0 要求 |
| --- | --- | --- | --- |
| `trackRepo.list({ sort, limit, offset })` | `get_tracks` | 已存在 | `sort` 支持 `title / artist / album / addedAt / lastPlayed`；默认过滤 `missingAt !== null`。 |
| `trackRepo.recentlyAdded(limit)` | `get_recently_added` | 已存在 | 返回非 missing 曲目，按 `addedAt DESC`。 |
| `trackRepo.byAlbum(albumId)` | `get_album_tracks` | 已存在 | 返回非 missing 曲目，按 disc/track/title 排序。 |
| `trackRepo.byArtist(artistId)` | `get_artist_tracks` | 已存在 | 返回非 missing 曲目，按 title 排序。 |
| `albumRepo.list()` | `get_albums` | 已存在 | 返回 `id/name/albumArtistName/year/coverPath/trackCount`。 |
| `albumRepo.get(albumId)` | `get_album` | 缺 repo/command，db 有 `albums::get_by_id` | v0.5.0 必须补齐，避免详情页拉全量专辑列表。 |
| `artistRepo.list()` | `get_artists` | 已存在 | 当前返回 `id/name/addedAt/updatedAt`，不含 `trackCount`；列表页曲目数如要显示必须新增后端字段，否则不显示。 |
| `artistRepo.get(artistId)` | `get_artist` | 缺 repo/command，db 有 `artists::get_by_id` | v0.5.0 必须补齐，避免详情页拉全量艺人列表。 |
| `playlistRepo.list()` | `get_playlists` | 已存在 | 返回 `id/name/description/coverPath/trackCount`。 |
| `playlistRepo.tracks(playlistId)` | `get_playlist_tracks` | 已存在但只返回 `Track[]` | v0.5.0 必须改为返回 `PlaylistTrack[]`，包含 `playlistPosition`。 |
| `playlistRepo.create(name, description?)` | `create_playlist` | 已存在 | v0.5.0 必须 trim name，空名返回 `InvalidInput`。 |
| `playlistRepo.rename(playlistId, name)` | `rename_playlist` | 已存在 | v0.5.0 必须 trim name，空名返回 `InvalidInput`。 |
| `playlistRepo.delete(playlistId)` | `delete_playlist` | 已存在 | 删除后刷新 `["playlists"]` 并跳转 `/playlists`。 |
| `playlistRepo.addTrack(playlistId, trackId)` | `add_to_playlist` | 已存在 | 允许重复添加同一曲目；返回新增 `playlistPosition`。 |
| `playlistRepo.removeTrack(playlistId, trackId, playlistPosition)` | `remove_from_playlist` | 已存在 | 删除后后端必须压实 position 为 `0..n-1`。 |
| `playlistRepo.reorder(playlistId, sourcePosition, destinationPosition)` | `reorder_playlist` | 已存在但语义需收紧 | 两个 position 都是 0-based 服务端 position；后端单事务重排为 `0..n-1`。 |
| `playerRepo.play({ trackId, queueTrackIds, queueIndex })` | `player_play` | 已存在 | `queueTrackIds[queueIndex]` 必须等于 `trackId`；队列只包含可播放曲目。 |

新增 TypeScript 类型：

```ts
export interface PlaylistTrack extends Track {
  playlistPosition: number;
}
```

`playlistPosition` 是服务端 `playlist_tracks.position` 压实后的 0-based 值。UI 不用数组 index 当作持久身份；所有移除和排序 mutation 都使用最近一次 query 返回的 `playlistPosition`。

### 6.2 播放列表排序语义

v0.5.0 保留现有 position-based command，但必须把语义写死并修正后端洞位问题：

```ts
playlistRepo.reorder(
  playlistId: number,
  sourcePosition: number,
  destinationPosition: number,
): Promise<void>
```

规则：

1. `sourcePosition` 和 `destinationPosition` 均为 0-based 服务端 position，不是过滤后的临时数组 index。
2. `destinationPosition` 表示移动完成后的最终 position。
3. 服务端必须在一个事务内完成移动，并把该播放列表所有 position 压实为 `0..n-1`。
4. `sourcePosition` 越界、`destinationPosition` 越界、播放列表不存在时返回错误且不改变顺序。
5. `remove_from_playlist` 删除后也必须压实 position，避免后续拖拽基于有洞 position 计算错误。
6. 前端排序 mutation pending 时禁用新的拖拽；失败后 invalidate `["playlistTracks", playlistId]` 回到服务端顺序。

之所以不采用 `trackId` 作为唯一移动标识，是因为当前 schema 允许同一首歌在同一播放列表中出现多次，`trackId` 不能唯一定位一行。

### 6.3 Missing Track 规则

v0.3.0 扫描器对消失文件使用软删除：设置 `missing_at`，不物理删除曲目，避免破坏 playlist 和 play_history 引用。v0.5.0 UI 必须显式处理。

规则：

1. 普通资料库列表隐藏 missing 曲目：
   - `trackRepo.list`
   - `trackRepo.recentlyAdded`
   - `trackRepo.byAlbum`
   - `trackRepo.byArtist`
2. 播放列表详情保留 missing 曲目，因为它们仍是用户列表的一部分。
3. missing 曲目在播放列表详情中置灰，显示“文件缺失”短标记。
4. missing 曲目禁止播放，不进入 `queueTrackIds`。
5. missing 曲目允许从播放列表移除。
6. 如果播放列表全部曲目都是 missing，“播放”按钮禁用。

### 6.4 React Query key

统一 query key：

```ts
["tracks", sort]
["recentlyAdded", limit]
["albums"]
["albumTracks", albumId]
["artists"]
["artistTracks", artistId]
["playlists"]
["playlistTracks", playlistId]
```

扫描完成时使用 predicate 刷新资料库相关 query，不能只刷新顶层列表：

```ts
queryClient.invalidateQueries({
  predicate: (query) =>
    [
      "tracks",
      "recentlyAdded",
      "albums",
      "albumTracks",
      "artists",
      "artistTracks",
      "playlistTracks",
    ].includes(String(query.queryKey[0])),
});
```

`playlistTracks` 也要刷新，因为扫描可能把播放列表中的曲目标记为 missing 或重新发现；刷新后 missing 曲目仍保留显示，只改变灰态和可播放状态。

播放列表变更时刷新：

```ts
["playlists"]
["playlistTracks", playlistId]
```

创建、重命名、删除、添加、移除、排序都必须使用 mutation pending 状态禁用重复提交。快速连续操作后，以服务端返回/刷新结果为准，不能长期展示旧数据。

### 6.5 Repository 边界

继续遵守项目原则：只有 `src/repositories/` 允许调用 `invoke()`。v0.5.0 可补充 TypeScript 参数类型，但不把 Tauri 调用散落到页面或组件。

v0.5.0 允许小范围补后端 command 和 db 查询来满足 API 对照表，但不得改变 v0.4.0 播放引擎线程模型。

## 7. 视觉与交互约束

v0.5.0 的视觉目标是“像可用的桌面音乐资料库”，不是完整 Apple Music 复刻。

具体约束：

1. 主布局使用左侧固定宽度侧边栏，宽度约 240px。
2. 侧边栏和 MiniPlayer 使用半透明背景 + `backdrop-filter`；macOS 原生 `NSVisualEffectView` 留到后续版本评估。
3. v0.5.0 不做背景动效、封面动态背景、复杂转场或大屏视觉 polish。
4. 不使用大面积单一紫色/蓝紫渐变，不使用装饰性光球或 bokeh 背景。
5. 封面缺失时使用稳定的多色渐变占位，颜色根据 id 或标题哈希生成。
6. 表格行高度稳定，hover、选中态、拖拽态不改变布局尺寸。
7. 按钮使用 `lucide-react` 图标，文本按钮只用于明确动作，例如“创建”“删除”“重命名”。
8. 文本必须在窄宽度下截断或换行，不能盖住按钮或 MiniPlayer。
9. 桌面优先，但窗口宽度低于 760px 时侧边栏可折叠为顶部/底部紧凑导航。

## 8. 错误处理

前端错误处理规则：

- 查询失败：页面级 `ErrorState` 显示错误文本和重试按钮。
- 播放失败：沿用 `MiniPlayer` 的 `playback_error` 展示；列表页不重复弹多个错误。
- 创建/重命名播放列表失败：对话框内显示错误，不关闭对话框。
- 删除播放列表失败：保留当前页并显示错误。
- 添加曲目到播放列表失败：菜单内显示失败状态，允许重试。
- 排序失败：刷新播放列表曲目，回到服务端顺序。
- 点击 missing 曲目播放：按钮禁用；如果通过键盘触发行操作，也直接返回，不调用 `player.play()`。

空状态规则：

- 空资料库：引导用户到设置或调用已有添加文件夹动作。
- 空播放列表：提示“从歌曲列表添加曲目”，不显示不可用的排序控件。
- 空专辑/艺人详情：显示“没有可播放曲目”，不自动跳转。
- 播放列表只有 missing 曲目：显示曲目行和移除操作，但播放按钮禁用。

## 10. 手动验收

用真实扫描资料库验收：

1. 启动 app 后默认进入最近添加页面。
2. 侧边栏能进入歌曲、专辑、艺人、播放列表、设置。
3. 歌曲页点击任意曲目能播放，下一首按当前排序推进。
4. 专辑页能进入详情，播放按钮从第一首开始播放整张专辑。
5. 艺人页能进入详情，曲目列表播放队列只包含该艺人曲目。
6. 能创建一个播放列表并从歌曲页添加曲目。
7. 播放列表详情能播放、移除曲目、拖拽排序；排序失败后刷新回服务端顺序。
8. 重命名后侧边栏和详情页同步更新。
9. 删除播放列表后不会留下失效路由。
10. 扫描完成后资料库页面数据刷新。
11. 空资料库和空播放列表显示明确下一步。
12. MiniPlayer 始终可见，列表内容不会被底栏遮挡。
13. missing 曲目在资料库页隐藏，在播放列表详情中置灰、禁播、可移除。
14. 快速连续创建 / 重命名 / 删除播放列表时，不出现长期旧数据闪回。

## 11. 推荐实施顺序

1. Repository API audit：对照第 6.1 节确认入参、返回字段、缺失 command，并先补 `albumRepo.get`、`artistRepo.get`、`PlaylistTrack.playlistPosition`、播放列表 position 压实。
2. AppShell + Sidebar + 路由重构：先让 `/library/recent`、`/library/songs`、`/library/albums`、`/library/artists`、`/playlists`、`/settings` 都能进入正确页面。
3. `TrackTableView` + `TrackTable` + Recent/Songs：先跑通“列表 -> 当前排序队列播放”主链路。
4. Albums/Artists + 详情页：使用 `albumRepo.get`、`artistRepo.get`，避免详情页拉全量列表。
5. 播放列表 CRUD：创建、重命名、删除、侧边栏刷新和删除后跳转。
6. 播放列表详情：带 `playlistPosition` 的曲目、missing 灰态、移除、拖拽排序和失败刷新。
7. 扫描完成刷新、空/错/加载态、并发 mutation pending 状态补齐。
8. 测试、门禁和 debug app 构建。

`/playlists` 总览页是低优先级入口，先做紧凑列表即可；不要先投入卡片视觉细节。v0.5.0 的主要验收价值在 Sidebar、资料库页面、详情页和 PlaylistDetail。
