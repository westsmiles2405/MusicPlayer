# MusicPlayer v0.6.0 搜索、收藏与最近播放 · 设计文档

- 创建日期：2026-05-02
- 版本：v0.6.0
- 文档状态：Draft v1.0
- 基于：v0.5.0 library playlists（tag `v0.5.0`）

---

## 1. 背景

v0.5.0 已完成资料库浏览、专辑/艺人详情、播放列表 CRUD、播放列表详情和与音频引擎的队列联动。产品已经具备“扫描本地音乐、浏览资料库、整理播放列表、播放歌曲”的基础能力，但还缺少“快速找到内容”和“回到常听内容”的日常使用闭环。

根据 `CLAUDE.md` 当前版本路线与 v0.5.0 设计文档，v0.6.0 的版本目标明确为：

1. 全局搜索
2. 收藏
3. 最近播放

这三个能力都已经有一部分底层条件：

- Rust 侧已具备 SQLite FTS5 搜索基础能力。
- 后端已有 `set_favorite` 和播放历史记录相关命令。
- v0.5.0 已经有 `TrackTable`、React Query、嵌套路由、AppShell 和播放队列调用约定。

因此 v0.6.0 不应该变成新的视觉大版本，而应该成为一个“搜索中枢型”版本：在不改动既有信息架构主骨架的前提下，把搜索、收藏、最近播放接入现有资料库和播放闭环。

## 2. 目标 / 非目标

### 2.1 目标

1. 新增一级搜索入口，支持跨歌曲、专辑、艺人、播放列表的全局搜索。
2. 新增“收藏的歌曲”页面，作为用户长期偏好的稳定入口。
3. 新增“最近播放”页面，作为用户短期回访入口。
4. 搜索、收藏、最近播放都必须复用现有播放队列体系，支持从列表直接播放。
5. 搜索结果中的歌曲必须支持加入播放列表，并展示正确的收藏状态。
6. 收藏、播放历史写入、扫描完成后，React Query 缓存刷新必须可预测且可验收。
7. 继续坚持 `src/repositories/` 作为唯一 IPC 边界，页面和组件不直接调 Tauri command。

### 2.2 非目标

- 不做 Now Playing 大屏。
- 不做 `⌘F` 快捷键。
- 不做主题跟随系统强化。
- 不做 i18n 扩展或英文文案补全。
- 不做首页聚合页。
- 不做搜索建议、最近搜索、复杂筛选、排序面板或高级相关性 UI。
- 不做新的拖拽交互或新的视觉系统重构。

## 3. 版本定位

v0.6.0 的产品定位是：

**搜索 + 收藏 + 最近播放 + 队列闭环**

它的职责不是增强“看起来像 Apple Music”的视觉层，而是让 MusicPlayer 从“能浏览资料库的播放器”升级为“能在日常使用里快速找到、标记和回访内容的播放器”。

视觉上保持 v0.5.0 的结构和风格，只做小范围补充，不引入会把版本拖散的全屏背景、复杂转场、Now Playing overlay 或主题系统改造。相关增强统一留给 v0.7.0。

## 4. 信息架构

### 4.1 侧边栏

在 v0.5.0 的 `Sidebar` 基础上新增三个入口：

1. `搜索`
2. `收藏`
3. `最近播放`

推荐顺序：

1. 搜索
2. 最近添加
3. 歌曲
4. 专辑
5. 艺人
6. 收藏
7. 最近播放

播放列表分组和设置入口保持原样，不为 v0.6.0 重新设计导航骨架。

### 4.2 路由

正式采用以下新增路由：

```text
/search
/library/favorites
/library/recent-plays
```

继续保留：

```text
/library/recent
/library/songs
/library/albums
/library/albums/:id
/library/artists
/library/artists/:id
/playlists
/playlists/:id
/settings
```

注意：v0.6.0 正式统一使用 `/library/recent-plays`，不再混用 `recently-played`。

## 5. 数据与接口设计

### 5.1 Repository 边界

前端继续遵守以下规则：

- 只有 `src/repositories/` 可以调用 Tauri `invoke()`
- 页面和组件只消费 repository 接口
- React Query 负责查询缓存和失效

v0.6.0 不新增越过 repository 的任何捷径。

### 5.2 搜索契约

搜索首版只定义一个简单、稳定的聚合结果，不引入复杂排序配置。

```ts
interface SearchResult {
  tracks: Track[];
  albums: Album[];
  artists: Artist[];
  playlists: Playlist[];
}
```

repository 接口目标：

```ts
search(query: string, limitPerGroup?: number): Promise<SearchResult>
```

规则：

- `query` 为空或纯空白时不发请求。
- 首版不做搜索建议、搜索历史或自动补全。
- 首版不暴露相关性调节参数。
- `limitPerGroup` 表示每一类结果的最大数量，不是总结果数。
- `limitPerGroup` 默认值为 `10`，由 repository 或命令层兜底。

### 5.3 收藏契约

收藏页只做“喜欢的歌曲”。

推荐新增：

```ts
favoriteRepo.list(): Promise<Track[]>
favoriteRepo.set(trackId: number, favorite: boolean): Promise<void>
```

也可以复用现有 `trackRepo` / `libraryRepo` 命名风格实现，只要最终页面拿到的是 `Track[]`，并能与 `TrackTable` 直接兼容。

### 5.4 最近播放契约

最近播放页展示“按曲目去重”的最近播放结果，而不是明细流。

页面语义采用：

- 每首歌只显示最近一次播放
- 按最近播放时间倒序排列

推荐接口形状：

```ts
interface RecentPlayedTrack extends Track {
  lastPlayedAt: number;
}

recentPlayRepo.list(limit?: number): Promise<RecentPlayedTrack[]>
```

如果后端仍保留明细 history 接口，可以继续保留，但页面层应消费“去重后的最近播放列表”，避免把“同一首歌连续播放三次”直接渲染成三行。

如果实现阶段决定 v0.6.0 首版不展示最近播放时间，可以临时降级为 `Track[]`，但页面不得渲染不存在的时间字段。默认推荐仍然是返回 `RecentPlayedTrack[]`，因为它能更稳定地表达“去重后的最近播放时间”。

### 5.5 DTO 与命令命名

为避免 implementation plan 阶段继续猜测命令和数据结构，v0.6.0 固定以下命名：

前端 repository：

```ts
searchRepo.search(query: string, limitPerGroup?: number): Promise<SearchResult>
favoriteRepo.list(): Promise<Track[]>
favoriteRepo.set(trackId: number, favorite: boolean): Promise<void>
recentPlayRepo.list(limit?: number): Promise<RecentPlayedTrack[]>
```

Tauri command：

```text
library_search_all
library_get_favorite_tracks
library_get_recent_played_tracks
```

Rust DTO：

```rust
SearchResultDto {
    tracks: Vec<TrackView>,
    albums: Vec<AlbumView>,
    artists: Vec<ArtistView>,
    playlists: Vec<PlaylistView>,
}
```

如果实现中发现现有项目命名需要与已有 `get_*` 风格统一，可以做等价调整，但 implementation plan 必须围绕一个固定命名版本展开，不能同时保留两套候选命名。

## 6. 页面设计

### 6.1 SearchPage

路径：`/search`

组成：

1. 顶部搜索输入框
2. 下方搜索结果区

结果区按四类分组：

1. 歌曲
2. 专辑
3. 艺人
4. 播放列表

展示规则：

- 输入为空：显示静态空状态，引导用户输入关键词。
- 查询中：显示 loading 状态。
- 查询失败：显示错误状态与重试按钮。
- 查询成功但无结果：显示“未找到相关内容”空状态。

交互规则：

- 歌曲结果复用 `TrackTable`
- 专辑/艺人/播放列表结果用轻量列表或卡片
- 点击专辑/艺人/播放列表结果进入对应详情页
- 歌曲结果支持：
  - 直接播放
  - 加入播放列表
  - 展示收藏状态并切换收藏

### 6.2 FavoritesPage

路径：`/library/favorites`

数据源：

- 收藏歌曲列表

展示规则：

- 复用 `TrackTable`
- 支持播放、加入播放列表、取消收藏
- 空状态文案应明确告诉用户：可以在歌曲、专辑详情、艺人详情或搜索结果中标记喜欢

### 6.3 RecentPlaysPage

路径：`/library/recent-plays`

数据源：

- 去重后的最近播放曲目列表

展示规则：

- 复用 `TrackTable`
- 支持播放和加入播放列表
- 如果 repository 返回 `RecentPlayedTrack[]`，页面可显示 `lastPlayedAt`
- 如果实现首版临时降级为 `Track[]`，则不显示时间字段
- 空状态文案应强调：播放任意歌曲后，这里会出现记录

## 7. 组件复用与新增边界

### 7.1 复用

v0.6.0 应优先复用 v0.5.0 已有能力：

- `AppShell`
- `Sidebar`
- `PageHeader`
- `LoadingState`
- `ErrorState`
- `EmptyState`
- `TrackTable`
- `AlbumGrid`
- `ArtistList`

### 7.2 新增

首版建议只新增搜索页需要的轻量结果组件，例如：

- `SearchInput`
- `SearchSection`
- `SearchResultList`

它们应尽量薄，不复制 `TrackTable` 已经承担的表格和播放逻辑。

### 7.3 TrackTable 扩展

v0.6.0 允许扩展 `TrackTable`，但不得复制一套搜索专用表格。

推荐新增 props：

```ts
showFavorite?: boolean;
onToggleFavorite?: (track: Track, favorite: boolean) => void;
```

如需额外展示列，也允许增加轻量扩展能力，例如：

```ts
extraColumns?: TrackTableExtraColumn[];
```

规则：

- 收藏状态展示和切换通过页面注入 `onToggleFavorite`
- `TrackTable` 不直接调用 Tauri command
- `TrackTable` 也不直接依赖 repository
- 搜索页、收藏页以及未来其它需要收藏操作的页面都复用这一扩展，而不是复制一套表格组件

## 8. 播放与队列规则

v0.6.0 所有新增页面都必须接入现有队列约定。

### 8.1 搜索结果中的歌曲

播放时使用当前搜索结果中的“可播放歌曲子集”作为队列。

规则：

- `missingAt !== null` 的曲目不可播放
- 搜索结果歌曲的队列只由当前搜索结果歌曲列表构成
- 不把专辑、艺人、播放列表分组内容混入歌曲队列

### 8.2 收藏页

播放时使用当前收藏歌曲列表中的可播放曲目作为队列。

### 8.3 最近播放页

播放时使用当前最近播放列表中的可播放曲目作为队列。

## 9. 缓存与失效规则

这一部分是 v0.6.0 的正式验收项，不是实现细节备注。

### 9.1 收藏切换成功后

至少失效以下 query：

- `["favoriteTracks"]`
- `["tracks"]`
- `["albumTracks", albumId]`
- `["artistTracks", artistId]`
- `["search", query]`

说明：

- 如果当前上下文没有 `albumId` / `artistId` / `query`，对应失效可以由页面按上下文决定是否触发。
- 目标是避免“歌曲页已喜欢，但搜索结果仍显示未喜欢”的假 bug。

### 9.4 收藏切换的 optimistic update

v0.6.0 推荐对收藏切换使用 optimistic update，以保证交互及时反馈。

规则：

- 用户点击收藏后，允许先更新当前列表中的收藏状态
- mutation 失败时必须 rollback 到 mutation 前状态
- mutation 成功后按本节规则 invalidate 相关 query
- 禁止只改本地 UI 而不落库
- 如果某些页面不适合 optimistic update，也必须在实现时给出明确理由，不能默认忽略 rollback

### 9.2 播放历史写入成功后

至少失效：

- `["recentPlays"]`

如果当前页面还展示播放次数、最近播放时间等附属状态，也可以额外失效相关曲目 query，但 `recentPlays` 是最低要求。

### 9.3 扫描完成后

至少失效：

- `["search"]`
- `["favoriteTracks"]`
- `["recentPlays"]`
- `["tracks"]`
- `["albums"]`
- `["artists"]`

如果扫描影响播放列表中的 missing 状态，也允许补充：

- `["playlistTracks", playlistId]`

但这不是 v0.6.0 新设计的核心变化，而是沿用已有扫描刷新原则。

## 10. 错误处理

### 10.1 搜索

- 输入为空不报错，不请求
- 查询失败显示错误状态与重试按钮
- 不因为某一类结果为空而整体失败

### 10.2 收藏

- 收藏切换失败时，应恢复按钮状态并提示错误
- 不允许页面展示与服务端持久状态长期不一致
- 收藏切换默认采用 optimistic update；失败时 rollback 到 mutation 前状态，并显示错误提示

### 10.3 最近播放

- 历史读取失败时显示错误状态
- 最近播放为空时显示空状态，不视为错误

## 11. 测试策略

### 11.1 前端

至少覆盖：

1. 搜索页：
   - 空查询不请求
   - 查询后四类结果分组渲染
   - 歌曲结果能触发播放
2. 收藏页：
   - 列表渲染
   - 取消收藏后触发 query invalidation
3. 最近播放页：
   - 列表渲染
   - 空状态渲染
4. repository：
   - 搜索返回结果形状正确
   - 收藏与最近播放调用正确命令

### 11.2 Rust / 后端

至少覆盖：

1. 搜索命令返回四类结果的契约
2. 收藏查询接口返回正确曲目列表
3. 最近播放去重接口按最近时间倒序输出

## 12. 验收标准

v0.6.0 完成时，必须满足以下条件：

1. 侧边栏中存在“搜索”“收藏”“最近播放”入口。
2. `/search` 可用，且能返回歌曲、专辑、艺人、播放列表四类结果。
3. 搜索结果中的歌曲可直接播放，并接入现有队列体系。
4. 搜索结果中的歌曲可加入播放列表。
5. 收藏页展示“喜欢的歌曲”，并支持取消收藏。
6. 最近播放页展示“按曲目去重、按最近播放时间倒序”的结果。
7. 收藏状态切换后，相关页面和搜索结果不会长期显示旧状态。
8. 播放历史写入后，最近播放页能够反映最新变化。
9. 页面和组件不直接调用 IPC，所有命令仍通过 `src/repositories/`。
10. `pnpm test`、`pnpm build`、Rust 对应新增测试通过。

## 13. 与后续版本的边界

v0.6.0 完成后，下一阶段 v0.7.0 再处理：

- Now Playing 大屏
- Apple Music 风格视觉 polish
- `⌘F` 和媒体相关快捷键
- 主题跟随系统强化
- i18n 文案收敛

这样可以保持版本切分清晰：

- `v0.5.0`：资料库与播放列表
- `v0.6.0`：搜索、收藏、最近播放
- `v0.7.0`：视觉与交互强化
