## 1. 背景

v0.2.1 完成了核心数据层与质量修复 — 44 个 Rust 测试、2 个前端测试、TS 类型同步，`cargo fmt` / `clippy -D warnings` / `pnpm lint` / `pnpm build` 均已通过。`LibraryPage` 当前显示 `曲目: 0 / 专辑: 0 / 艺人: 0`，等待数据进入。

v0.3.0 的任务：**让用户选一个本地目录，把里面的音乐文件读进数据库，在资料库页面看到真实数据。** 这是阻塞 v0.4.0 音频引擎的关键路径 — 没有数据，播放器没东西可放。

## 2. 目标 / 非目标

### 2.1 目标

1. 用户能在 LibraryPage 空态或设置页添加一个/多个本地音乐文件夹
2. 后台扫描递归识别 `mp3 / m4a / flac / wav / aac` 文件，使用 lofty 读 tag、提取封面
3. 增量差分：二次扫描区分 added / updated / unchanged / missing
4. 文件移动检测：通过内容 hash 识别"文件路径变了但内容没变"
5. 进度条 + 错误汇总反馈给前端
6. 扫描写入后 FTS5 搜索结果与曲目元数据保持一致
7. 端到端冒烟：选用户真实音乐目录 → LibraryPage 显示非 0 数据 + 至少 1 张封面

### 2.2 非目标（推迟）

- 文件监听器（`notify` crate） → v0.3.1
- 启动后自动监听目录变化 → v0.3.1
- 完整 Mac App Store 沙盒权限流程与 bookmark 恢复测试 → v0.9（MAS 打包前）
- 启动时自动重扫上次目录 → v0.3.1
- 扫描结果分享/导出 → 无计划
- iTunes XML 库导入 → v1.1+

### 3.2 后续添加 / 管理目录

入口：**设置页 → 音乐文件夹** 面板
- 已添加目录列表（路径 + "移除" 按钮）
- 底部：[+ 添加文件夹] / [↻ 重新扫描全部]
- "重新扫描全部" = 调用 `start_scan()` 扫所有根（fast-path 跳过未变文件；5k 曲目目标 < 5s，真实数值以 v0.3.0 手测为准）。v0.3.0 不做单根重扫，理由：UI 复杂度 / 命令面增大；按钮太多；fast-path 已经够快
- "移除目录"语义：mark_missing 该根下所有 tracks（`UPDATE tracks SET missing_at=?now WHERE root_folder_id=?id`）+ 解链 `SET root_folder_id=NULL` + 删 scan_folders 行。**不**物理删 tracks 行 — 用户可能后来重新添加，希望保留 favorite/play_count。该操作需要额外 db helper，不能复用 v0.2.1 的 `tracks::mark_missing(ids, now)`。

### 3.3 扫描中切页

进度条置于 MiniPlayer 上方（全局可见），用户可自由切到 Albums/Artists/Settings 等任意页面。任何时候可点取消。取消会让 scanner 在下一个文件边界停止（不会回滚已 commit 的事务）。

依赖变更：v0.3.0 若继续使用 `blake3` 做文件/封面 hash，需要在 `src-tauri/Cargo.toml` 增加 `blake3 = "1"`。否则必须改用现有 `xxhash-rust`，不要在代码里直接引用未声明依赖。

### 4.2 数据流（单次 scan）

```
commands::start_scan
   │
   ▼
spawn_blocking → scanner::scan_folders(folders, on_progress, abort_flag)
   │
   ├── 1. 多根去重（canonicalize → 排序 → 移除被父路径包含的子根）
   │
   ├── 2. walk_dir() → Vec<PathBuf>      (~瞬时, 万级文件 < 1s)
   │
   ├── 3. mpsc channel: walk → reader pool (4 workers)
   │      每 worker：lofty 读 tag → RawTrack → cover bytes 提取 → channel
   │
   ├── 4. indexer 单线程消费 channel：
   │      ┌─ 50 首一个事务（rusqlite Transaction）
   │      ├─ 每首 upsert_artist / upsert_album / upsert_track / cache_cover
   │      ├─ 每事务 commit 后 emit scan_progress（节流 100ms）
   │      └─ 单文件错误 → push ScanReport.errors，继续
   │
   ├── 5. 收尾：UPDATE tracks SET missing_at=now()
   │           WHERE last_seen_at < scan_id AND missing_at IS NULL
   │
   └── 6. emit scan_done(report)
```

## 5. 数据模型变更

### 5.1 V1 schema 复用（无需迁移的部分）

V1 已经为扫描器准备好了大部分字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `tracks.file_size` | INTEGER NOT NULL | B1 fast-path：判 `(mtime,size)` 未变 |
| `tracks.file_modified_at` | INTEGER NOT NULL | 同上 |
| `tracks.hash` | TEXT (nullable) | B1 步骤 5：文件移动检测 |
| `tracks.last_seen_at` | INTEGER NOT NULL | B2 增量差分：扫到的写当前 scan_id，扫完比对 mark_missing |
| `tracks.missing_at` | INTEGER (nullable) | 标记缺失（v0.2.1 已有 mark_missing 函数） |
| `scan_folders.{id, path, bookmark_data, last_scanned_at}` | - | 多根管理；`bookmark_data` v0.3.0 可先为空，MAS 沙盒前再启用恢复流程 |

→ **无需重复造列**。差分 SQL 直接用 `last_seen_at`。

### 5.2 V3 迁移（必须，v0.3.0 内）

**目的**：
1. 跨根隔离 — 扫 ~/Music 不应把 ~/Downloads 的 track 标 missing。
2. 修正 v0.2.1 的 FTS5 更新语义 — v0.2.1 的 `tracks_fts` 是普通 standalone FTS5 表，不是 contentless 表；`UPDATE/DELETE -> rebuild` 不能从 `tracks/albums/artists` 自动重建新内容。v0.3.0 会实际更新曲目元数据，因此必须在本版本修正 FTS 同步，不能推迟。

```sql
-- src-tauri/migrations/V3__scanner_support.sql
ALTER TABLE tracks ADD COLUMN root_folder_id INTEGER REFERENCES scan_folders(id);
CREATE INDEX idx_tracks_root_folder ON tracks(root_folder_id);
CREATE INDEX idx_tracks_hash ON tracks(hash);  -- B1 step 5 movedDetect 路径必需

DROP TRIGGER IF EXISTS trg_tracks_fts_insert;
DROP TRIGGER IF EXISTS trg_tracks_fts_update;
DROP TRIGGER IF EXISTS trg_tracks_fts_delete;
DROP TRIGGER IF EXISTS trg_albums_fts_update;
DROP TRIGGER IF EXISTS trg_artists_fts_update;

-- 保留 standalone FTS5 表，使用 row-level DELETE/INSERT 同步。
-- tracks_fts 不使用 content=''，所以可以普通 DELETE。
CREATE TRIGGER trg_tracks_fts_insert AFTER INSERT ON tracks
WHEN NEW.missing_at IS NULL
BEGIN
  INSERT INTO tracks_fts(rowid, title, album_name, artist_name)
  VALUES (
    NEW.id,
    NEW.title,
    (SELECT name FROM albums WHERE id = NEW.album_id),
    (SELECT name FROM artists WHERE id = NEW.primary_artist_id)
  );
END;

CREATE TRIGGER trg_tracks_fts_update AFTER UPDATE ON tracks
BEGIN
  DELETE FROM tracks_fts WHERE rowid = OLD.id;
  INSERT INTO tracks_fts(rowid, title, album_name, artist_name)
  SELECT
    NEW.id,
    NEW.title,
    (SELECT name FROM albums WHERE id = NEW.album_id),
    (SELECT name FROM artists WHERE id = NEW.primary_artist_id)
  WHERE NEW.missing_at IS NULL;
END;

CREATE TRIGGER trg_tracks_fts_delete AFTER DELETE ON tracks
BEGIN
  DELETE FROM tracks_fts WHERE rowid = OLD.id;
END;

-- 艺人/专辑重命名很少见，直接刷新受影响曲目的 FTS 行。
CREATE TRIGGER trg_albums_fts_update AFTER UPDATE OF name ON albums
WHEN NEW.name IS NOT OLD.name
BEGIN
  DELETE FROM tracks_fts
   WHERE rowid IN (SELECT id FROM tracks WHERE album_id = NEW.id);
  INSERT INTO tracks_fts(rowid, title, album_name, artist_name)
  SELECT t.id, t.title, NEW.name, ar.name
    FROM tracks t
    LEFT JOIN artists ar ON ar.id = t.primary_artist_id
   WHERE t.album_id = NEW.id AND t.missing_at IS NULL;
END;

CREATE TRIGGER trg_artists_fts_update AFTER UPDATE OF name ON artists
WHEN NEW.name IS NOT OLD.name
BEGIN
  DELETE FROM tracks_fts
   WHERE rowid IN (SELECT id FROM tracks WHERE primary_artist_id = NEW.id);
  INSERT INTO tracks_fts(rowid, title, album_name, artist_name)
  SELECT t.id, t.title, al.name, NEW.name
    FROM tracks t
    LEFT JOIN albums al ON al.id = t.album_id
   WHERE t.primary_artist_id = NEW.id AND t.missing_at IS NULL;
END;

-- 迁移末尾重建一次 standalone FTS 内容，清理 v0.2.1 旧触发器可能留下的脏数据。
DELETE FROM tracks_fts;
INSERT INTO tracks_fts(rowid, title, album_name, artist_name)
SELECT t.id, t.title, al.name, ar.name
  FROM tracks t
  LEFT JOIN albums al ON al.id = t.album_id
  LEFT JOIN artists ar ON ar.id = t.primary_artist_id
 WHERE t.missing_at IS NULL;
```

> 实施时需要先验证 SQLite FTS5 对 standalone 表的 `DELETE FROM tracks_fts WHERE rowid=?` 行为；若本机/CI SQLite 版本不接受，需要改成“scanner 每个事务后显式重建受影响行”的 helper，而不是回退到 `'rebuild'`。

**写时序**：
1. `scan_id = now_ms()` 在 scanner 入口生成
2. 每次 upsert_track：`last_seen_at = scan_id`，`root_folder_id = <当前根 id>`
3. 扫完后一句 SQL（仅作用于本次扫描的根）：

```sql
UPDATE tracks
   SET missing_at = ?now, updated_at = ?now
 WHERE last_seen_at < ?scan_id
   AND missing_at IS NULL
   AND root_folder_id IN (?...被扫描的根 id 列表);
```

> 老数据兼容：v0.2.1 测试用例造的 track 行 `root_folder_id` 为 NULL；不在 IN 列表里 → 不会被误标 missing。安全。

**模型同步要求**：V3 不只是 SQL 迁移。必须同步更新：
- Rust `Track` / `NewTrack` / `Track::from_row` / `TrackView` 相关 SELECT 列
- `tracks::insert`、`tracks::update_by_path`、moved-path 更新 helper
- TS `Track` 类型与前端 repository
- 相关测试 fixture / `make_basic_track`

## 6. 关键策略

### 6.1 文件身份判定（B1）

逐级 fast-path，从最便宜到最贵：

| 步骤 | 判定 | 动作 |
|---|---|---|
| 1 | `path` 命中（tracks.file_path UNIQUE） | → 跳到 2 |
| 2 | `(file_modified_at, file_size)` 与 db 一致 | **unchanged**：跳过 reread，仅更新 `last_seen_at = scan_id` |
| 3 | `(file_modified_at, file_size)` 不一致 | **updated**：lofty re-read，update_by_path（含 hash 重算） |
| 4 | `path` 未命中，算 hash | → 跳到 5 |
| 5 | `(hash, file_size)` 唯一命中（需要给 tracks.hash 加索引） | **moved**：`UPDATE tracks SET file_path=?, file_modified_at=?, file_size=?, last_seen_at=?, root_folder_id=?, missing_at=NULL WHERE id=?`，不重读 tag |
| 6 | `hash` 未命中 | **added**：full lofty read + insert |

如果 `(hash, file_size)` 命中多行（重复文件、同头部文件、用户复制同一首歌），不要做 moved 自动匹配，降级为 added / full metadata read，避免把播放次数和收藏错误迁移到另一个文件。

**hash 定义**：`blake3(file_bytes[0..min(64KB, file_size)])`，在后台线程内计算。理由：64KB 头通常包含 ID3v2 + 一段音频帧，足以做移动检测的快速启发式。如果不新增 `blake3` 依赖，则用现有 `xxhash-rust::xxh3`，但文档和实现必须一致。

### 6.2 多根去重（A2）

```rust
fn canonicalize_roots(roots: Vec<PathBuf>) -> Vec<PathBuf> {
    let mut canon: Vec<_> = roots.into_iter()
        .filter_map(|p| p.canonicalize().ok())
        .collect();
    canon.sort();  // 字典序：父路径排在前
    let mut result = Vec::new();
    for path in canon {
        if !result.iter().any(|r: &PathBuf| path.starts_with(r)) {
            result.push(path);
        }
    }
    result
}
```

测试用例：`["/Users/x/Music/Albums", "/Users/x/Music"]` → `["/Users/x/Music"]`

### 6.3 封面缓存（B3）

```rust
fn cache_cover(image_bytes: &[u8], cache_dir: &Path) -> AppResult<String> {
    let hash16 = &blake3::hash(image_bytes).to_hex()[..16];
    let ext = detect_image_ext(image_bytes).unwrap_or("bin");
    let filename = format!("{hash16}.{ext}");
    let path = cache_dir.join(&filename);
    if !path.exists() {
        std::fs::write(&path, image_bytes)?;
    }
    Ok(format!("covers/{filename}"))  // 相对 app_data_dir 根
}
```

存储位置：`app_data_dir().join("covers/")`（用 `tauri::path::PathResolver`）。
`albums.cover_path` 存相对路径 `covers/<hash16>.<ext>`，渲染时前端拼接 `convertFileSrc()`。不要把 PNG/WebP 强行写成 `.jpg`；要么保留原始字节并匹配扩展名，要么用 `image` crate 解码后统一转 JPEG。

### 6.4 并行度（C1）

```
walk_dir (1 thread) ──┐
                      │  mpsc<PathBuf> (cap 256)
                      ▼
            reader pool (4 workers, spawn_blocking)
                      │  mpsc<Result<RawTrack, ScanError>> (cap 64)
                      ▼
              indexer (1 thread, holds the only writer Connection)
                      │
                      ▼
                   SQLite
```

`Result` 让 reader 单文件错误（lofty 解析失败、IO 错）顺着管道流到 indexer，由 indexer 决定 push 进 `ScanReport.errors[]` 还是写库。indexer 本身在 upsert 时也可能产生错误（外键违反等），同样 push 而不 panic。

- reader 是 CPU + I/O 混合（lofty parse + cover bytes 复制），4 worker 经验值
- indexer **必须**单线程：rusqlite `Connection` 不 `Send` 到多线程，且 SQLite 写并行无收益
- channel 容量是 backpressure：reader 太快会被阻塞，避免内存爆

### 6.5 进度节流（C2）

```rust
struct ProgressEmitter {
    last_emit: Instant,
    interval: Duration,  // 100ms
}

impl ProgressEmitter {
    fn maybe_emit(&mut self, app: &AppHandle, p: &ScanProgress) {
        if self.last_emit.elapsed() >= self.interval {
            app.emit("scan_progress", p).ok();
            self.last_emit = Instant::now();
        }
    }
    fn force_emit(&mut self, app: &AppHandle, p: &ScanProgress) {
        app.emit("scan_progress", p).ok();
        self.last_emit = Instant::now();
    }
}
```

收尾时强制 emit 最终帧 + emit `scan_done` 才能保证前端进度条到 100%。

## 7. 模块详细设计

### 7.1 `library/scanner.rs`

```rust
pub struct ScanProgress {
    pub done: usize,
    pub total: usize,
    pub current_file: Option<String>,
}

pub struct ScanReport {
    pub added: usize,
    pub updated: usize,
    pub moved: usize,
    pub unchanged: usize,
    pub missing: usize,
    pub errors: Vec<ScanError>,
}

pub struct ScanError {
    pub path: String,
    pub message: String,
}

pub struct AbortFlag(Arc<AtomicBool>);

pub struct ScanContext<'a> {
    pub db: &'a Database,
    pub cache_dir: PathBuf,
}

pub fn scan_folders(
    ctx: ScanContext<'_>,
    folders: &[PathBuf],
    abort: AbortFlag,
    on_progress: impl Fn(ScanProgress) + Send + 'static,
) -> AppResult<ScanReport>;
```

边界：
- scanner 不直接 emit 事件，由调用方（`commands::start_scan`）通过 `on_progress` 桥接
- `abort` 在每个文件循环开头 check `.load(Ordering::Relaxed)`，停止时返回当前 `ScanReport`（部分结果保留）
- `cache_dir` 由 command 层通过 `app.path().app_data_dir().join("covers")` 传入，scanner 不自己猜路径

### 7.2 `metadata/reader.rs`

```rust
pub struct RawTrack {
    pub path: PathBuf,
    pub hash: String,             // 16-char hex
    pub mtime_ms: i64,
    pub size_bytes: i64,
    pub title: Option<String>,
    pub artists: Vec<String>,     // primary + featuring
    pub album: Option<String>,
    pub album_artist: Option<String>,
    pub track_no: Option<i32>,
    pub disc_no: Option<i32>,
    pub year: Option<i32>,
    pub genre: Option<String>,
    pub duration_ms: i64,
    pub bitrate: Option<i32>,
    pub sample_rate: Option<i32>,
    pub channels: Option<i32>,
    pub codec: Option<String>,

}

pub fn read_track(path: &Path) -> AppResult<RawTrack>;
```

无 tag 兜底：`title = file_stem`，`album = "Unknown Album"`，`album_artist = "Unknown Artist"`，`artists = ["Unknown Artist"]`。这些常量要复用，避免大小写/语言差异导致重复艺人或专辑。

### 7.4 `library/indexer.rs`

```rust
pub struct UpsertOutcome {
    pub kind: UpsertKind,  // Added / Updated / Moved / Unchanged
    pub track_id: i64,
}

pub enum UpsertKind { Added, Updated, Moved, Unchanged }

pub fn upsert_track(
    txn: &Transaction,
    raw: &RawTrack,
    cover_relpath: Option<&str>,
    scan_id: i64,
    root_folder_id: i64,
) -> AppResult<UpsertOutcome>;
```

内部步骤：判定 outcome（B1 流程）→ upsert_artist (primary) → upsert_album（含 cover_path）→ upsert_track（写 `last_seen_at = scan_id`、`root_folder_id`、清 `missing_at`）→ link 其他 artists（feature/composer 在 v0.3.0 仅写 `track_artists` 表，UI 暂不展示）。

事务边界：indexer 每 50 首开启一个外层事务。v0.2.1 的 `tracks::insert` 内部会自己开 `unchecked_transaction()`，不能直接嵌套在这层事务里。v0.3.0 需要二选一：
- 增加 `insert_in_tx` / `update_by_path_in_tx` / `mark_present_in_tx` 等 helper，复用外层 transaction。
- 或取消 50 首外层事务，让每个 repo 自己提交。实现更简单但较慢。

本 spec 推荐第一种，避免扫描大库时每首歌一次 commit。

### 7.5 `commands/library.rs`

| 命令 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `add_music_folder` | `path: String` | `ScanFolder` | 追加进 scan_folders，返回新行 |
| `remove_music_folder` | `id: i64` | `()` | 单事务：① mark_missing 该根 tracks ② `UPDATE tracks SET root_folder_id=NULL WHERE root_folder_id=?id` ③ `DELETE FROM scan_folders WHERE id=?id` |
| `list_music_folders` | - | `Vec<ScanFolder>` | |
| `start_scan` | - | `()` | fire-and-forget，spawn_blocking 启动 |
| `cancel_scan` | - | `()` | 设 abort flag |

事件：
- `scan_progress` → `ScanProgress`
- `scan_done` → `ScanReport`

命名迁移：v0.2.1 已有占位 `add_folder(path) -> ()`。v0.3.0 应替换为 `add_music_folder(path) -> ScanFolder`，或保留 `add_folder` 作为兼容 alias，但前端新代码统一使用 `libraryRepo.addFolder()` 包装 `add_music_folder`。不要同时让两个命令写入不同语义的 `scan_folders`。

并发约束：`start_scan` 检查"已有扫描进行中" → 返回 `AppError::Busy`（前端按钮 disable）。扫描状态需要放在 Tauri managed state 中，例如 `ScanManager { running: AtomicBool, abort: Arc<AtomicBool> }`；不要用全局 static，方便测试和多窗口生命周期管理。

### 7.6 前端

```ts
// src/repositories/libraryRepo.ts
export interface ScanFolder { id: number; path: string; addedAt: number; lastScannedAt: number | null }
export interface ScanProgress { done: number; total: number; currentFile: string | null }
export interface ScanReport { added: number; updated: number; moved: number; unchanged: number; missing: number; errors: ScanError[] }

export const libraryRepo = {
  addFolder: (path: string) => invoke<ScanFolder>("add_music_folder", { path }),
  removeFolder: (id: number) => invoke<void>("remove_music_folder", { id }),
  listFolders: () => invoke<ScanFolder[]>("list_music_folders"),
  startScan: () => invoke<void>("start_scan"),
  cancelScan: () => invoke<void>("cancel_scan"),
};

// src/hooks/useScanProgress.ts
export type ScanPhase = "idle" | "scanning" | "done";
export function useScanProgress(): {
  phase: ScanPhase;
  progress: ScanProgress | null;
  report: ScanReport | null;
  cancel: () => void;
};
```

LibraryPage 行为：
- `phase === "idle" && tracks.data?.length === 0` → 显示空态卡片 + "添加音乐文件夹"按钮
- `phase === "scanning"` → 显示扫描中视图
- 否则 → 正常资料库（待 v0.5.0 UI 全套，本期先列表）

## 8. 错误处理

| 严重度 | 场景 | 处理 |
|---|---|---|
| 轻微 (log) | 单文件 lofty 解析失败 / cover 写盘失败 | push `ScanReport.errors[]`，scan 继续 |
| 中等 (toast) | 扫描完成但 errors > 0 | toast「添加 N，3 个文件失败」，可点开详情列表 |
| 严重 (modal) | 用户选的目录不存在 / 无读权限 / 数据库 corrupt | `AppError::Scan(String)` → 前端 ErrorBoundary 模态 |

`AppError` 在 v0.2.1 已有 `NotFound` 和 `Scan(String)`，v0.3.0 只需要新增 Busy：
```rust
#[error("scan already in progress")]
Busy,
```

## 11. 风险与回退

| 风险 | 影响 | 缓解 |
|---|---|---|
| v0.2.1 FTS5 UPDATE/DELETE 同步语义不可靠 | updated / moved / missing 后搜索结果可能陈旧 | v0.3.0 的 V3 迁移必须修正 row-level FTS 同步，并加 updated/missing 搜索测试 |
| lofty 对 ALAC / 异常 FLAC 解析失败 | 个别文件读不进来 | 单文件错误隔离，记入 errors，不阻断 |
| 大库一次事务 OOM | indexer 内存爆 | 50 首一个 commit，channel cap 控制 backpressure |
| 内容 hash IO 慢 | 万级文件扫描数十秒 | 只 hash 前 64KB |
| 跨平台路径大小写 / NFC vs NFD（macOS） | 同一文件被识别成两个 path | `Path::canonicalize()` 在 macOS 返回 NFD；保存前 `to_string_lossy()` 入库一致 |
| 用户加超大根目录（~/）误扫 | 扫描卡几小时 | 不设软限制，但 walkdir 跳过隐藏目录（`.git`、`.Trash` 等）+ 文档警告 |
| 取消扫描时部分事务已 commit | 数据半成品 | 设计上接受：扫描随时可恢复；取消时不要更新 `scan_folders.last_scanned_at`，下次重扫重新校正 |
